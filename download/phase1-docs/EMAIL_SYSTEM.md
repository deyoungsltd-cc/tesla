# Email System Requirements

## 1. Email Architecture

The platform's email system is built on a modern, queue-based architecture designed for high deliverability, template consistency, and operational reliability. **Resend** serves as the primary email delivery provider, chosen for its developer-friendly API, robust webhook infrastructure, and strong deliverability reputation with major email providers. **React Email** is used for all email template development, enabling the creation of dynamic, data-driven email templates using familiar React component patterns and TypeScript for type safety. All outgoing emails are routed through a **BullMQ** job queue backed by **Redis**, ensuring that email sending is asynchronous, retried on transient failures, and decoupled from the main application request lifecycle. This architecture means that API responses to user actions (such as registration or deposit submission) are never delayed by email delivery operations. The email service runs as a dedicated worker process that can be scaled independently of the main application servers. The system supports both immediate transactional delivery for time-sensitive emails and scheduled delivery for marketing communications, with clear separation between the two pathways.

## 2. Email Categories

### Transactional Emails (Immediate)

Transactional emails are triggered by specific user actions and system events that require immediate notification. These emails are sent with the highest priority in the queue and are expected to be delivered within seconds of the triggering event. The platform generates transactional emails for the following events: a **welcome email** sent upon successful registration with platform overview and next steps; an **email verification OTP** containing the one-time code required to confirm the user's email address; a **password reset OTP** sent when a user requests a password change; a **2FA setup confirmation** confirming that two-factor authentication has been enabled on the account; a **login alert** sent when the system detects a login from a new device or unrecognized IP address; **deposit confirmed** notifications when a cryptocurrency or gift card deposit is verified and credited to the user's wallet; **withdrawal processed** confirmations when a withdrawal request has been completed and funds sent; **investment matured** notifications when an investment plan reaches its duration and returns are credited; and **commission credited** alerts when referral or binary bonus commissions are deposited into the user's wallet. Transactional emails cannot be disabled by the user, as they are essential for account security and financial awareness.

### Security Emails (Immediate)

Security emails form a critical subcategory of transactional communications focused exclusively on account protection and threat awareness. These emails are sent immediately upon detection of security-relevant events and are designed to alert users to potential unauthorized access or account changes. Security emails include **password changed** notifications (sent whenever a user's password is modified, whether initiated by the user or through a reset flow); **2FA enabled/disabled** confirmations whenever the two-factor authentication status changes; **suspicious activity alerts** when the system detects patterns consistent with account compromise, such as multiple failed login attempts, logins from geographically distant locations within a short timeframe, or unusual transaction patterns; **account locked** notifications when an account is automatically locked due to security triggers or manually locked by an administrator; and **session revoked** notifications when active sessions are terminated, either by the user through the security settings page or by an administrator. Security emails are delivered with the highest priority and include clear, actionable instructions for the user to follow if they did not initiate the reported activity.

### Marketing Emails (Scheduled)

Marketing emails are non-essential communications sent to users who have explicitly opted in to receive promotional content. These emails are generated on a scheduled basis, not in response to real-time events, and are sent during optimized delivery windows to maximize open rates. Marketing email types include **newsletter** communications containing platform updates, market insights, and educational content about investment strategies; **promotional offers** highlighting special bonuses, limited-time commission increases, or other incentives; and **new plan announcements** notifying users of newly available investment tiers, updated return rates, or changes to existing plans. Marketing emails are only sent to users who have an active opt-in status on their communication preferences. Users can opt out at any time through the preference center or the one-click unsubscribe link included in every marketing email. The system tracks opt-in and opt-out status per user and ensures that no marketing emails are sent to unsubscribed users, in compliance with CAN-SPAM, GDPR, and other applicable regulations.

### System Emails (Immediate)

System emails are operational communications that keep users informed about the status of their interactions with platform support and administrative processes. These include **support ticket updates** notifying users when their ticket receives a response, is escalated, or is closed; **KYC status change** notifications informing users when their verification submission is approved, rejected, or requires additional information; and **gift card deposit status** updates informing users when their gift card submission has been received, is under review, has been verified and credited, or has been rejected with a reason. System emails bridge the gap between real-time in-app notifications and the need for a persistent, accessible record of important status changes. Users can reference these emails later to review the history of their interactions with the platform's administrative processes.

## 3. Template System

All email templates are built using **React Email**, which provides a component-based approach to email template development that is fully compatible with the project's existing React and TypeScript stack. Templates are organized as React components that accept typed props for dynamic data, ensuring compile-time validation of the data passed to each template. A shared layout wrapper component provides consistent branding across all emails: a header section displaying the platform logo and a branded color bar using the platform's signature red and black color scheme, and a footer section containing the platform's legal name, physical address, and an unsubscribe link for marketing emails. All templates are designed with a responsive layout that renders correctly across email clients including Gmail, Outlook, Apple Mail, and mobile email applications. The template system uses inline CSS styles (auto-generated by React Email) to ensure maximum compatibility with email clients that strip out `<style>` tags. Every email template includes a plain text fallback version that is automatically generated and used by email clients that do not support HTML rendering. The template library is version-controlled alongside the application code, and changes to templates follow the same pull request and review process as application code changes.

## 4. Email Queue

All outgoing emails are processed through a **BullMQ** job queue backed by **Redis**, providing reliable, asynchronous email delivery that is decoupled from the main application request cycle. When any part of the application needs to send an email, it creates a job in the email queue with the template identifier, recipient address, dynamic data, and priority level. BullMQ processes these jobs using dedicated worker processes that communicate with the Resend API to deliver the emails. Failed deliveries are automatically retried using an exponential backoff strategy: the first retry occurs after 1 minute, the second after 5 minutes, and the third and final retry after 15 minutes. If all three retries fail, the job is moved to a **dead letter queue** where it remains for inspection and manual intervention. The administration dashboard includes a dedicated email queue monitoring interface that displays the current queue depth, processing rate, failed jobs, and dead letter queue contents. Administrators can view the details of failed email jobs (including the error message from Resend), manually retry individual failed jobs, or purge the dead letter queue. This queue-based architecture ensures that temporary issues with the Resend API, network connectivity, or email provider rate limits do not result in lost communications.

## 5. Delivery Monitoring

Email delivery is monitored comprehensively through **Resend's webhook system**, which provides real-time event data on the status of every sent email. The platform subscribes to webhooks for the following event types: **delivered** (confirmed receipt by the recipient's mail server), **bounced** (permanent delivery failure), **complained** (recipient marked the email as spam), **opened** (recipient opened the email, if tracking pixels are supported), and **clicked** (recipient clicked a link within the email). All webhook events are logged to the PostgreSQL database with the email ID, recipient, event type, timestamp, and any additional metadata provided by Resend. The monitoring system calculates aggregate delivery metrics including delivery rate, bounce rate, and complaint rate, and triggers automated alerts when any metric exceeds defined thresholds. Specifically, if the bounce rate exceeds 5% or the complaint rate exceeds 0.1% over a rolling 24-hour period, an alert is sent to the administrative team for investigation. High bounce or complaint rates can indicate deliverability issues that may affect the platform's sender reputation and, consequently, the deliverability of all future emails. The monitoring dashboard provides both real-time metrics and historical trend analysis to help the team identify and resolve delivery issues proactively.

## 6. Unsubscribe Management

The platform implements a comprehensive unsubscribe management system that complies with GDPR, CAN-SPAM, and other applicable email marketing regulations. Every marketing email includes a prominent one-click unsubscribe link in the email footer that immediately updates the user's opt-in status in the database. The unsubscribe process does not require the user to log in or navigate to a preference center; a single click on the unsubscribe URL is sufficient to opt out. Upon unsubscribing, the user receives a confirmation email acknowledging the change, and no further marketing emails are sent to that address. The preference center, accessible from the user's account settings, provides granular control over communication preferences, allowing users to opt in or out of specific marketing categories (newsletters, promotional offers, plan announcements) independently. **Transactional emails cannot be unsubscribed**, as they are essential for account security, financial awareness, and regulatory compliance. The system clearly distinguishes between transactional and marketing emails in the preference center, and users are informed that transactional emails will continue regardless of their marketing preferences. For users covered by GDPR (identified by their registered country being within the European Economic Area), the system maintains a record of when and how consent was obtained, and users can exercise their right to withdraw consent at any time.

## 7. Sender Configuration

Email deliverability depends heavily on proper sender domain configuration and authentication. The platform uses a custom verified domain for sending all emails, ensuring that emails appear to come from the platform's own domain rather than a generic third-party sender address. The DNS configuration for the sending domain includes **SPF** (Sender Policy Framework) records that authorize Resend's mail servers to send emails on behalf of the platform's domain, **DKIM** (DomainKeys Identified Mail) records that add cryptographic signatures to outgoing emails proving the email was not tampered with in transit, and **DMARC** (Domain-based Message Authentication, Reporting, and Conformance) records that instruct receiving mail servers on how to handle emails that fail SPF or DKIM checks. For Phase 1, the platform uses Resend's shared IP infrastructure with a dedicated sending domain, which provides good deliverability for moderate email volumes. As email volume grows, migration to a dedicated IP address for transactional emails is planned to isolate transactional delivery reputation from marketing email patterns. The reply-to address for all transactional and system emails is set to the platform's support inbox, enabling users to reply directly to email notifications and have their response routed to the support ticket system. Marketing emails use a separate reply-to address or a no-reply address to manage response volume appropriately.

## 8. Multi-language Emails

The platform's global user base requires that email communications be delivered in each user's preferred language. The email system detects the user's language preference from their account profile settings and selects the appropriate localized template variant for each email. During user registration, the system captures the user's preferred language either through explicit selection or by inferring it from the browser's Accept-Language header. This preference is stored in the user's profile and can be changed at any time through the account settings. The template system supports multiple language variants through a parameterized approach: each email template has a base structure shared across all languages, with translatable strings externalized into language-specific resource files. When an email is generated, the user's language preference is passed to the template renderer, which loads the corresponding translation resources and populates the template with the correct language content. The translation management system tracks which email templates have been translated into which languages, and any template that lacks a translation for a user's preferred language falls back to English (the platform's default language). The system supports right-to-left (RTL) text rendering for languages such as Arabic, with the template layout automatically adjusting direction based on the language. Initial language support covers English, Spanish, French, German, Portuguese, Arabic, Chinese (Simplified), and Japanese, with additional languages added based on user demand and market expansion.
