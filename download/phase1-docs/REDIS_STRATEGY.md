# Redis Strategy

## 1. Redis Overview

Redis 7+ serves as the platform's primary in-memory data store, underpinning several critical subsystems including application-level caching, session management, rate limiting, job queuing, OTP (one-time password) storage, and real-time publish/subscribe messaging. Redis was selected for its sub-millisecond latency, rich data structure support (strings, hashes, sorted sets, lists, and streams), and mature ecosystem including BullMQ for job queue management. For Phase 1, the platform deploys a **single Redis instance** to simplify operations and reduce infrastructure complexity while the user base and transaction volume are within manageable bounds. The architecture is designed with forward compatibility in mind, using key naming conventions and access patterns that can be migrated to a **Redis Cluster** topology in future phases without application-level code changes. The single instance is configured with persistence enabled (both RDB and AOF) to prevent data loss on restart, and is deployed behind the application's internal network with no direct external access. All Redis connections are authenticated with a strong password and, where infrastructure permits, TLS-encrypted. The platform's Redis instance is monitored for memory usage, connection count, command latency, and eviction statistics through a dedicated monitoring dashboard.

## 2. Caching Strategy

The platform implements a **cache-aside** pattern as the default caching strategy across all cached data types. In this pattern, the application first checks Redis for the requested data; if the data is present and not expired (based on the key's TTL), it is returned directly without querying PostgreSQL. If the data is not in Redis (a cache miss), the application queries PostgreSQL, returns the result to the caller, and asynchronously populates the Redis cache with the fetched data for subsequent requests. This approach ensures that the application always functions correctly even if Redis is unavailable, falling back to direct database queries as described in the Failure Handling section. The following specific cache types are implemented with tailored TTLs and invalidation strategies.

**Investment plans cache** stores the current configuration of all four investment plan tiers (Basic, Silver, Gold, Platinum) including deposit ranges, durations, return rates, and active/inactive status. This data is read on every plan browsing request and changes infrequently, making it an ideal caching candidate. The TTL is set to 5 minutes, balancing freshness with cache hit rate. The cache is explicitly invalidated whenever an Admin modifies any plan configuration through the admin panel, ensuring that plan changes are reflected immediately rather than waiting for TTL expiration.

**User session data cache** stores metadata associated with active user sessions, including the user ID, device identifier, IP address at login time, and the refresh token expiration timestamp. The TTL is configured to match the JWT refresh token expiry period (typically 7 to 30 days, depending on the session type), and the key is automatically cleaned up when the session expires or is explicitly revoked. Session cache misses trigger re-authentication, so this cache is not a performance optimization but rather the authoritative store for session metadata.

**Exchange rate cache** stores the latest cryptocurrency-to-USD exchange rates fetched from an external pricing API. Since cryptocurrency prices fluctuate continuously, the cache TTL is set to 60 seconds, providing near-real-time rates while preventing excessive API calls to the external provider. The cache is populated on a scheduled basis by a background worker that fetches current rates every 60 seconds, ensuring that the cache is always warm and users never experience a cache miss that triggers a synchronous external API call.

**Static content cache** stores system-wide configuration values and feature flags that are used across the application but change very rarely. This includes platform settings (platform name, support email, maintenance mode status), fee configurations (withdrawal fee percentage, minimum withdrawal amount), and feature flags that control the availability of new features. The TTL is set to 10 minutes, and explicit invalidation occurs whenever an Admin modifies any system setting. This cache significantly reduces database load for configuration reads, which occur on nearly every API request.

## 3. Session Storage

Redis serves as the authoritative store for user session metadata, complementing the stateless JWT-based authentication system. When a user logs in, the application generates a refresh token and stores associated metadata in Redis as a hash structure keyed by the session token. The stored metadata includes the user ID for quick token-to-user resolution, the device identifier (user agent hash) to support multi-device session management and new-device detection, the IP address at the time of login for security alerting, and the absolute expiration timestamp for the session. The session key's TTL in Redis is set to match the refresh token's expiration, ensuring that expired sessions are automatically cleaned up by Redis without requiring a separate cleanup process. The system also maintains a secondary index of active sessions per user using a Redis set keyed by `sessions:{userId}`, enabling the application to quickly enumerate all active sessions for a user when displaying the session management UI in security settings. When a user explicitly logs out or revokes a session, the corresponding key is deleted from Redis, immediately invalidating the refresh token. User online status is derived from session data: a user is considered "online" if they have at least one session key in Redis that has been accessed (refreshed) within the last 5 minutes, tracked via a separate TTL-extended key.

## 4. Rate Limiting

The platform implements **sliding window rate limiting** using Redis sorted sets, providing precise control over request rates without the burst-allowing behavior of fixed-window approaches. Each rate-limited endpoint has a defined maximum number of requests allowed within a sliding time window. The rate limit key is constructed from the endpoint identifier and the client identifier, which can be the user's ID (for authenticated endpoints) or the client's IP address (for unauthenticated endpoints like login and registration). For each incoming request, the current timestamp is added to the sorted set, and all entries older than the sliding window are removed. If the remaining count in the sorted set exceeds the configured limit, the request is rejected with a `429 Too Many Requests` response that includes headers indicating the limit, the remaining requests allowed, and the time at which the limit will reset. The following rate limit configurations are applied: authentication endpoints (login, password reset, email verification) allow 5 requests per minute per IP to prevent brute-force attacks; API endpoints allow 60 requests per minute per authenticated user for general use; withdrawal and deposit endpoints allow 10 requests per minute per user to prevent rapid repeated submissions; and KYC submission endpoints allow 3 requests per hour per user to prevent verification spam. These limits are configurable through the system settings without requiring code deployment, enabling rapid response to abuse patterns.

## 5. OTP Storage

One-time passwords for email verification, password reset, and 2FA are stored in Redis with strict security and timing controls. OTP values are **hashed using SHA-256** before storage; the raw OTP is never stored in plaintext in any data store. The Redis key for each OTP includes the OTP type (email verification, password reset, 2FA) and the target identifier (user ID or email address), enabling independent rate limiting and management per OTP type. Each OTP key has a **TTL of 10 minutes**, after which the OTP expires automatically and cannot be verified. Alongside the OTP value, the system stores an **attempt counter** as a separate key with the same TTL, tracking how many times the user has submitted an incorrect OTP. If the attempt counter exceeds 5 for a single OTP, the OTP is immediately invalidated by deleting its key, and a new OTP must be requested. A separate **rate limit key** tracks OTP request frequency: users are limited to 3 OTP requests per 5 minutes per OTP type, preventing abuse of the OTP generation endpoint to flood a user's email inbox or to brute-force the OTP value. When a user successfully verifies an OTP, the key is immediately deleted to prevent replay attacks. The combination of hashing, short TTLs, attempt limits, and request rate limiting provides a robust OTP security model.

## 6. Job Queues (BullMQ)

BullMQ leverages Redis as its backing store to provide reliable, persistent job queues for asynchronous task processing. Each queue is configured with specific concurrency limits and priority levels to ensure that system resources are allocated appropriately based on task importance and processing requirements.

**Email queue** handles all outgoing email delivery jobs with a concurrency of 5 workers processing jobs simultaneously. This queue operates at **normal priority** and is the highest-volume queue in the system, processing transactional, security, system, and marketing emails. Jobs in this queue include the template identifier, recipient data, and dynamic template props. Failed jobs follow the standard retry policy with exponential backoff.

**Withdrawal processing queue** handles the sequential processing of approved withdrawal requests, including blockchain transaction signing and broadcasting for cryptocurrency withdrawals. This queue operates at **high priority** with a concurrency of exactly **1 worker**. The single-worker constraint is a deliberate financial safety measure: processing withdrawals sequentially ensures that balance checks and transaction ordering are deterministic, preventing race conditions where concurrent processing could allow a user to withdraw more than their available balance. Each withdrawal job locks the user's balance record during processing and releases it only after the blockchain transaction is confirmed or the job fails.

**Report generation queue** handles the asynchronous creation of administrative reports, analytics summaries, and export jobs. This queue operates at **low priority** with a concurrency of **2 workers**, reflecting that report generation is not time-sensitive and should not compete with financial or user-facing operations for system resources. Report jobs may involve complex database queries and data aggregation, so keeping concurrency low prevents database overload during report generation.

**Gift card verification queue** handles the processing and routing of gift card deposit submissions, including image analysis and admin notification routing. This queue operates at **normal priority** with a concurrency of **3 workers**. The concurrency level allows reasonable throughput for gift card processing while keeping the review workload manageable for administrators.

**Cache invalidation queue** handles delayed and batched cache invalidation operations that are triggered by data changes but do not need to execute synchronously. This queue operates at **high priority** with a concurrency of **5 workers**. High priority ensures that cache invalidation propagates quickly to prevent stale data from being served to users. Common jobs in this queue include purging plan caches after configuration changes, clearing user-specific caches after profile updates, and invalidating exchange rate caches when the pricing source reports significant price movements.

## 7. Pub/Sub

Redis pub/sub provides the real-time messaging backbone for the platform's notification delivery system. The pub/sub architecture uses dedicated channels for different notification streams, ensuring that messages are routed efficiently to only the interested subscribers.

The primary pub/sub channel follows the naming pattern `notifications:{userId}`, where each authenticated user with an active SSE or WebSocket connection subscribes to their personal notification channel. When the application creates a notification for a user, it publishes the notification payload to that user's channel, and the real-time connection layer delivers it to the connected client immediately. This per-user channel pattern ensures that each client only receives notifications intended for them, eliminating the need for client-side filtering.

The admin event broadcasting channel (`admin:events`) is used for real-time delivery of administrative notifications to all connected admin users. When an admin-relevant event occurs (new KYC submission, withdrawal request, support ticket escalation), the event is published to this channel, and all admin clients subscribed to the channel receive the event simultaneously. This channel supports the admin dashboard's live operational view and ensures that admins are immediately aware of pending tasks.

The system alerts channel (`system:alerts`) is used for infrastructure and operational alerts that need to be broadcast to all connected clients or all admin clients. This channel is used for notifications such as scheduled maintenance announcements, where a single publish operation needs to reach all active users to inform them of the upcoming downtime. The pub/sub system does not persist messages, meaning that clients that are not connected at the time of publication will not receive the message; these clients will receive the notification through the standard polling fallback or when they next load the notification page, which queries the persistent notification store in PostgreSQL.

## 8. Cache Invalidation

Cache invalidation follows a dual strategy combining **explicit invalidation** on known data changes with **TTL-based expiration** as a safety net. Explicit invalidation is the primary mechanism and is triggered by application code whenever a write operation modifies data that is cached. For example, when an Admin updates an investment plan's return rate, the plan cache invalidation job is immediately dispatched to the cache invalidation queue, which deletes the relevant cache keys. This ensures that the next read request for plan data will fetch the updated values from PostgreSQL and populate the cache with fresh data. Explicit invalidation targets are specific keys or key patterns (using Redis `DEL` or `SCAN` + `DEL` for pattern-based invalidation), minimizing unnecessary cache churn. TTL-based expiration serves as a fallback to handle edge cases where explicit invalidation might be missed due to application bugs or direct database modifications. The TTL values for each cache type are intentionally kept short enough (5 minutes for plans, 10 minutes for settings) that stale data is quickly expired even without explicit invalidation. **Cache stampede protection** (preventing multiple concurrent requests from simultaneously repopulating a cold cache) is not implemented in Phase 1 because the expected concurrent write volume is low and the cost of occasional duplicate database queries during cache repopulation is negligible. If stampede protection becomes necessary at higher scale, it can be added using Redis-based distributed locking around cache population.

## 9. Persistence

Redis is configured with dual persistence mechanisms to balance data durability with performance. **RDB (Redis Database) snapshots** are configured to execute every 15 minutes, creating point-in-time snapshots of the entire Redis dataset. RDB snapshots are compact, single-file backups that are fast to create and restore, making them ideal for disaster recovery scenarios. However, RDB snapshots alone can lose up to 15 minutes of data in the event of a sudden failure. To mitigate this, **AOF (Append Only File)** persistence is also enabled with the `everysec` fsync policy, which appends every write command to the AOF log and flushes it to disk once per second. This configuration means that in a worst-case failure scenario, at most one second of data is lost. The AOF file is automatically rewritten by Redis when it grows beyond a configured size threshold, keeping the file compact and efficient for restart recovery. The maximum memory policy is set to **allkeys-lru**, which means that when Redis reaches its memory limit, it evicts the least recently used keys across all data types. This policy is appropriate for a caching-heavy workload where cached data can always be regenerated from PostgreSQL, but care is taken to ensure that critical keys (session tokens, OTPs, rate limit counters) have appropriate TTLs that prevent them from being the victims of LRU eviction under memory pressure.

## 10. Memory Planning

Accurate memory planning is essential to prevent Redis from exceeding its allocated memory and triggering eviction of important data. The following estimates are based on the expected Phase 1 workload of approximately 10,000 registered users with 1,000 concurrent active sessions.

**Caching data** is estimated to consume approximately 50–100MB, driven primarily by the investment plans cache (negligible size), exchange rate cache (small key set, frequent updates), static content cache (dozens of keys with small values), and user-specific caches that accumulate as users browse the platform. **Session data** is estimated at 20–30MB, based on approximately 1,000 concurrent sessions with each session hash consuming roughly 200–300 bytes. **Rate limiting data** is estimated at 10–20MB, with each rate limit key being a sorted set that accumulates timestamps within the sliding window. **OTP storage** is estimated at under 5MB at any given time, as OTPs are short-lived (10-minute TTL) and each entry is small. **BullMQ queue data** is estimated at 20–50MB, depending on the current depth of the email queue and other queues. **Pub/sub** has negligible memory impact as Redis pub/sub does not persist messages. Based on these estimates, the **minimum recommended memory allocation for production is 512MB**, with a **recommended allocation of 1GB** to provide comfortable headroom for traffic spikes, queue depth increases, and temporary data accumulation. The Redis `maxmemory` directive should be set to the allocated limit minus a 10% safety margin to prevent the operating system from killing the Redis process due to out-of-memory conditions.

## 11. Key Naming Convention

A consistent key naming convention is essential for operational clarity, debugging efficiency, and future migration to Redis Cluster (which requires key-based slot routing). All Redis keys follow a **prefix-based hierarchical naming scheme** using colons as delimiters: `{category}:{entity}:{identifier}`.

The following prefix categories are defined and used throughout the application:

- **`cache:{entity}:{id}`** — Application-level caches. Examples: `cache:plans:all` for the investment plans collection, `cache:rates:btc-usd` for the BTC/USD exchange rate, `cache:settings:platform` for platform configuration values.
- **`session:{userId}:{deviceId}`** — User session metadata. Example: `session:user_12345:device_abcde` storing the session hash with user ID, device info, IP, and expiry.
- **`ratelimit:{type}:{identifier}`** — Rate limiting counters. Examples: `ratelimit:login:192.168.1.1` for login attempts by IP, `ratelimit:api:user_12345` for general API rate limiting by user.
- **`otp:{type}:{identifier}`** — OTP storage and attempt tracking. Examples: `otp:email_verify:user_12345` for email verification codes, `otp:password_reset:user_12345` for password reset codes, `otp:attempts:email_verify:user_12345` for the attempt counter.
- **`queue:{name}`** — BullMQ internal keys (these are auto-generated by BullMQ but follow predictable patterns). Example: BullMQ creates keys like `bull:email:waiting`, `bull:email:active`, `bull:email:completed`.
- **`pubsub:{channel}`** — Pub/sub channel names (these are not stored keys but follow the convention for consistency). Examples: `pubsub:notifications:user_12345`, `pubsub:admin:events`, `pubsub:system:alerts`.

This convention ensures that related keys are grouped together visually in Redis monitoring tools, enables pattern-based key operations (such as invalidating all plan caches with `DEL cache:plans:*`), and provides clear context when inspecting Redis during debugging or operations.

## 12. Failure Handling

The platform is designed to degrade gracefully when Redis becomes unavailable, ensuring that core platform functionality remains operational even during Redis outages. The failure handling strategy is differentiated by the criticality of each Redis-dependent subsystem.

For **non-critical caches** (investment plans, exchange rates, static settings), the application falls back to direct PostgreSQL queries when a Redis connection cannot be established or when a Redis command times out. This fallback is transparent to the user; the only impact is increased database load and slightly higher response latency, as every request that would normally be served from cache now hits the database. The cache-aside pattern inherently supports this fallback because the application always has the database as the source of truth.

For **session management**, a Redis failure means that refresh token validation cannot be completed through the normal metadata lookup. In this scenario, the application falls back to validating the JWT refresh token's cryptographic signature and embedded claims (user ID, expiry) without the additional Redis metadata checks (device verification, IP verification). This degrades security slightly (device-bound session checks are bypassed during the outage) but maintains user access. Once Redis is restored, session metadata is re-populated on the next token refresh.

For **rate limiting**, a Redis failure results in rate limiting being temporarily disabled. The application logs a warning when rate limit checks cannot be performed and continues processing the request. This is an acceptable tradeoff because denying all requests during a Redis outage would cause a complete platform outage, which is worse than the temporary absence of rate limiting. Rate limiting is re-enabled automatically when Redis connectivity is restored.

For **BullMQ job queues**, a Redis failure causes all queue processing to pause. Workers detect the connection loss and enter a waiting state, periodically attempting to reconnect. Jobs that were in progress at the time of failure are not lost; BullMQ's design ensures that in-progress jobs are returned to the waiting state if the worker disconnects without completing them. When Redis is restored, workers reconnect, and job processing resumes from where it left off. The system implements a **circuit breaker pattern** for Redis dependencies: after a configurable number of consecutive connection failures, the circuit breaker opens and the application stops attempting Redis connections for a cooldown period, preventing repeated connection attempts from degrading application performance. When the cooldown expires, the circuit breaker allows a single probe connection; if it succeeds, the circuit closes and normal Redis operations resume.
