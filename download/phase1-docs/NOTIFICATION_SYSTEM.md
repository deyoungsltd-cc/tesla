# Notification System Requirements

## 1. In-App Notification System

The in-app notification system provides users with a persistent, real-time awareness of important events on the platform without requiring them to actively check their email. The system is accessible through a **notification bell icon** positioned prominently in the application header, which displays a badge showing the count of unread notifications. Clicking the bell icon opens a **notification center dropdown** that presents the most recent notifications in reverse chronological order, with unread notifications visually distinguished from read ones through a bold indicator and a colored dot. Each notification in the dropdown displays a concise title, a brief summary text, a timestamp indicating when the notification was generated, and an optional action link that navigates the user to the relevant section of the application (for example, clicking a deposit notification navigates to the wallet page). The dropdown loads the 10 most recent notifications and includes a "View All" link that navigates to the full **notification detail page**. The notification detail page presents the complete notification history organized by date, with pagination loading 20 notifications per page. Users can **mark individual notifications as read** by clicking on them, or use a **"Mark All as Read"** action to bulk-clear the unread status. Users can **delete individual notifications** or bulk-delete all read notifications to keep their notification history manageable. A **notification preferences** panel allows users to control which notification types they receive in-app, independent of their email notification preferences.

## 2. Notification Types

### Financial Notifications

Financial notifications keep users informed about all monetary activity on their account, ensuring transparency and timely awareness of fund movements. The system generates a **deposit received** notification when a cryptocurrency transaction is confirmed on the blockchain or a gift card deposit is verified and credited to the user's wallet, including the amount, currency, and transaction reference. A **withdrawal processed** notification is sent when a withdrawal request has been completed and the funds have been dispatched, including the amount sent, the destination address, and the network fee if applicable. An **investment return credited** notification is generated when an active investment plan reaches its maturity duration and the principal plus returns are credited to the user's wallet, showing the original investment amount, the return amount, and the total credited. A **commission earned** notification informs the user when referral commission or binary bonus income has been deposited into their wallet, including the commission amount, the source referral's username, and the commission tier applied. An **investment plan completed** notification provides a comprehensive summary when a plan finishes, including the plan tier, duration, deposited amount, return rate, and net profit. All financial notifications include a direct link to the relevant transaction or investment record in the user's history.

### Security Notifications

Security notifications are high-priority alerts designed to inform users of account-related security events that may require their attention or action. A **new device login** notification is triggered when the system detects a successful login from a device or IP address that has not been previously associated with the user's account, including the device type, approximate location (if available), and timestamp. A **password changed** notification confirms that the user's account password has been successfully modified, with a timestamp and a reminder that if the user did not initiate this change, they should contact support immediately. **2FA enabled/disabled** notifications are sent whenever the two-factor authentication status changes on the account, providing a clear record of security configuration changes. An **account locked** notification informs the user that their account has been locked due to multiple failed login attempts, administrative action, or security trigger detection, along with instructions for regaining access. Security notifications are always sent to both the in-app notification system and the user's registered email address, regardless of the user's notification preferences, because account security overrides user customization.

### Account Notifications

Account notifications cover changes to the user's account status, verification level, and profile-related reminders. A **KYC status change** notification is sent when the user's verification submission is approved (congratulating them and informing them of their new capabilities and limits) or rejected (explaining the reason and inviting them to resubmit). A **verification level upgraded** notification celebrates the user's progression through the KYC levels and clearly lists the new deposit limits, withdrawal limits, and plan tiers that have been unlocked. **Profile update reminders** are generated periodically if the user's profile information is incomplete or if their KYC documents are approaching expiration, nudging the user to take action to maintain full account access. These notifications help users stay aware of their account standing and take timely action to maintain uninterrupted access to platform features.

### Referral Notifications

Referral notifications keep users engaged with the platform's referral program by providing timely updates on their referral network activity. A **new referral signed up** notification informs the user when someone registers using their referral link, including the referral's username and the date of registration. A **referral made first deposit** notification is triggered when a referred user completes their first real deposit, indicating that the referrer is now eligible to start earning commissions. A **commission paid** notification confirms when referral commission has been calculated and credited to the user's wallet, breaking down the commission amount and the deposit or investment that generated it. A **binary bonus earned** notification informs the user when a binary bonus has been calculated based on their binary referral tree performance, including the bonus amount and the contributing legs. These notifications are critical for maintaining user engagement with the referral program, as they provide tangible evidence of the program's financial benefits and encourage continued referral activity.

### System Notifications

System notifications communicate platform-wide information that all users need to be aware of. **Scheduled maintenance** notifications are sent in advance of planned system maintenance windows, informing users of the expected downtime, the affected features, and the expected resolution time. **New features** notifications announce significant platform updates, new investment plan tiers, or enhanced functionality that users should be aware of. **Plan updates** notifications inform users of changes to existing investment plans, including modifications to return rates, duration periods, or deposit limits that may affect their investment decisions. System notifications are typically sent to all users simultaneously and are authored by administrators through the admin dashboard. They serve as the primary communication channel for platform-wide announcements that go beyond the scope of marketing emails.

## 3. Delivery Channels

Notifications are delivered through multiple channels to ensure users receive important information regardless of how they interact with the platform. The primary delivery channel is **in-app**, where notifications appear in real-time in the notification bell dropdown and notification center page. In-app delivery is instantaneous for users who are actively using the platform, leveraging real-time connection technologies described in the next section. The secondary delivery channel is **email**, used for critical notifications that users must receive even when they are not actively using the platform. Security notifications and high-value financial notifications (withdrawals, large deposits, investment maturities) are always sent via both in-app and email channels. A third delivery channel, **push notifications** (mobile browser and native app), is planned for a future development phase and is not implemented in Phase 1. The channel selection for each notification type is configurable in the notification preferences system, allowing users to customize their experience while ensuring that critical alerts always reach them through at least one channel.

## 4. Real-Time Delivery

Real-time notification delivery ensures that users see new notifications the moment they are generated, without needing to manually refresh the page. The system uses **Redis pub/sub** as the backbone for real-time notification distribution. When a notification is created, the application publishes it to a Redis channel specific to the target user (formatted as `notifications:{userId}`). Client-side JavaScript maintains a persistent connection that subscribes to this channel and receives notification events as they are published. The preferred transport mechanism for this connection is **Server-Sent Events (SSE)**, which provides a lightweight, unidirectional real-time channel from server to client that is well-supported by modern browsers and easier to implement than full-duplex WebSocket connections for a primarily server-push use case. If SSE is not available due to proxy or network configuration, the system falls back to **WebSocket** connections as a secondary transport. For clients that cannot maintain a persistent connection (for example, users behind restrictive corporate firewalls), the system implements a **polling fallback** that checks for new notifications every 30 seconds using a lightweight API endpoint. The polling endpoint returns only the count of new notifications since the last check and the most recent notification ID, minimizing bandwidth usage. When a polling client detects new notifications, it fetches the full notification data through a standard REST endpoint.

## 5. Notification Storage

Notifications are stored persistently in **PostgreSQL** to ensure they survive server restarts, remain accessible across multiple devices, and support the full notification history page. Each notification record includes the target user ID, notification type (categorized as financial, security, account, referral, or system), a title string, a body text, an optional action URL for navigation, a read/unread boolean flag, a soft-deleted flag, and timestamps for creation, read status change, and deletion. The system uses **soft delete** for notification removal, meaning that deleted notifications are flagged rather than physically removed from the database. This approach preserves notification history for auditing purposes and allows users to recover accidentally deleted notifications. An automated **purge process** runs daily and permanently deletes notifications that have been both read and soft-deleted for more than 90 days, preventing unbounded database growth. **Unread notifications are never auto-purged**, regardless of age, ensuring that users do not lose important information they have not yet seen. The notification detail page implements **server-side pagination** with 20 notifications per page, using cursor-based pagination for efficient querying of the notification table. The notification bell dropdown uses a separate lightweight query that fetches only the 10 most recent unread notifications plus enough recent read notifications to fill the display, minimizing the data transferred for the dropdown view.

## 6. Notification Preferences

The notification preferences system gives users granular control over which notifications they receive and through which channels. Each notification category (financial, security, account, referral, system) can be independently toggled on or off for both in-app and email delivery, giving users a 2x5 matrix of preference controls. The default state for all preferences is **enabled**, meaning new users receive all notification types through all available channels. Users can access their notification preferences through the account settings page, where each category is presented with clear descriptions and toggle switches. Security notifications are a special case: the in-app toggle for security notifications is always enabled and cannot be disabled by the user, though the email channel for security notifications can be toggled. This ensures that users always see security alerts in the app while giving them the option to reduce email volume. Notification preferences are stored in the user's profile record in PostgreSQL and are applied at the point of notification generation: before a notification is created and delivered, the system checks the user's preferences for the corresponding category and channel, and silently skips delivery to any disabled channel without generating an error or log entry. The preference system also supports a global "pause all notifications" toggle that disables all non-security in-app and email notifications, useful for users who are traveling or temporarily away from the platform.

## 7. Admin Notifications

Administrative users receive a separate, dedicated notification stream that is isolated from the standard user notification system. Admin notifications are generated by operational events that require administrative attention or awareness, and they are delivered only to users with Admin-level roles and above. The admin notification stream includes: **new KYC submissions** alerting admins that a user has submitted documents for verification review, with a direct link to the review queue; **withdrawal requests** notifying admins of pending withdrawal requests that require approval, including the amount and user details; **deposit verifications needed** alerting admins to cryptocurrency deposits or gift card submissions awaiting manual verification; **support ticket escalations** notifying admins when a support ticket has been escalated by an agent and requires higher-level attention; and **system alerts** covering operational issues such as high error rates, Redis connectivity problems, queue processing failures, or other infrastructure concerns. Admin notifications are displayed in a separate notification bell icon on the admin dashboard, with their own unread count and notification center. This separation ensures that operational alerts are not mixed with the admin's personal user notifications and can be prioritized appropriately. Admin notifications follow the same real-time delivery, storage, and pagination patterns as user notifications but are stored in a separate database table to enforce access control at the data level.

## 8. Badge & Count

The unread notification count displayed on the notification bell icon is a critical UX element that drives user engagement and ensures awareness of pending information. The system provides a dedicated **real-time unread count API endpoint** (`GET /api/notifications/unread-count`) that returns the total number of unread notifications for the authenticated user. This endpoint is optimized for performance, using a cached count stored in Redis that is updated atomically whenever a notification is created, marked as read, or deleted. The client-side application calls this endpoint on page load and subscribes to real-time count updates via the same SSE/WebSocket connection used for notification delivery. When a new notification arrives through the real-time channel, the client **optimistically increments** the displayed badge count immediately, before the API response confirms the new count, ensuring that the badge appears to update instantly. Periodically (every 60 seconds or on page navigation), the client re-fetches the actual count from the API endpoint to correct any drift caused by missed real-time events. The unread count includes all unread notifications across all categories, providing a single aggregate number. The count is visually displayed as a numeric badge on the bell icon when the count is between 1 and 99, and as "99+" when the count exceeds 99 to prevent the badge from becoming too large to display legibly. When the count is zero, the badge is hidden entirely, providing a clean visual state that indicates there is nothing new to review.
