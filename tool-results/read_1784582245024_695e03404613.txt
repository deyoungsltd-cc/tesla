    41→    41→## 1. Database Approach Overview
    42→    42→
    43→    43→### 1.1 Technology Stack
    44→    44→
    45→    45→| Component | Technology | Purpose |
    46→    46→|-----------|-----------|---------|
    47→    47→| Primary RDBMS | PostgreSQL 16+ | All persistent application data |
    48→    48→| ORM | Prisma ORM (v5+) | Type-safe database access, migrations, schema management |
    49→    49→| Caching / Ephemeral | Redis 7+ | Sessions, rate limits, exchange rate cache, job queues, OTP throttle |
    50→    50→| File Storage | Cloudinary | KYC documents, gift card screenshots |
    51→    51→| Email | Resend + React Email | Transactional and notification emails |
    52→    52→
    53→    53→### 1.2 Why PostgreSQL + Prisma
    54→    54→
    55→    55→**PostgreSQL** provides strict ACID compliance for financial data integrity, native `JSONB` support for flexible metadata columns, full-text search with `tsvector`/`tsquery` and GIN indexes, and mature tooling for backup (pg_dump + WAL archiving) and point-in-time recovery. These capabilities are essential for a financial platform handling investments, withdrawals, and audit trails.
    56→    56→
    57→    57→**Prisma ORM** delivers first-class PostgreSQL support with generated TypeScript types, an interactive transaction API (`prisma.$transaction()`) for multi-step financial operations, declarative migration management (`prisma migrate`), and a schema-first development workflow where the `schema.prisma` file serves as the single source of truth.
    58→    58→
    59→    59→### 1.3 Internal Accounting Currency
    60→    60→
    61→    61→USD is the sole internal accounting currency. All wallet balances, investment amounts, fee calculations, and commission values are stored in USD. Fiat display currencies (EUR, GBP, etc.) are handled at the presentation layer via real-time conversion using cached exchange rates. Cryptocurrency deposit amounts are converted to USD at the time of blockchain confirmation.
    62→    62→
    63→    63→### 1.4 Dual-Mode Data Separation
    64→    64→
    65→    65→Every user operates in two parallel modes: **Demo** and **Live**. Data separation is achieved through:
    66→    66→
    67→    67→- **Wallets**: Each user has exactly two wallets — one `demo`, one `live` — differentiated by the `type` field.
    68→    68→- **Investments**: The `mode` field on `UserInvestment` tags each investment as demo or live.
    69→    69→- **Transactions**: The `walletId` foreign key implicitly links each transaction to the correct mode via the wallet.
    70→    70→- **Deposits/Withdrawals**: The `mode` field explicitly tags each operation.
    71→    71→- **Referral Commissions**: Commissions from live deposits credit the sponsor's live wallet; demo commissions credit the demo wallet.

    75→    75→## 2. Schema Design Principles
    76→    76→
    77→    77→### 2.1 Normalization
    78→    78→
    79→    79→The schema adheres to **Third Normal Form (3NF)** as the baseline. Controlled denormalization is applied only where query performance demands it:
    80→    80→
    81→    81→- **Wallet balances** (`balance`, `availableBalance`, `lockedBalance`) are denormalized summaries maintained transactionally. The invariant `balance = availableBalance + lockedBalance` is always enforced application-side.
    82→    82→- **BinaryNode volumes** (`volumeLeft`, `volumeRight`) are incrementally updated denormalized counters to avoid recursive tree traversal on every query.
    83→    83→
    84→    84→### 2.2 UUID Primary Keys
    85→    85→
    86→    86→All entities use `UUID v4` as their primary key, generated application-side via `crypto.randomUUID()`. This prevents information leakage (no sequential ID enumeration), eliminates cross-environment ID collisions, and supports future distributed architectures.
    87→    87→
    88→    88→### 2.3 Soft Deletes
    89→    89→
    90→    90→All entities include a nullable `deletedAt` `DateTime` column. Hard `DELETE` is never used in application code. All queries include `WHERE deletedAt IS NULL` enforced through Prisma middleware. Hard deletes are reserved for GDPR erasure compliance after a mandatory holding period.
    91→    91→
    92→    92→### 2.4 Timestamps
    93→    93→
    94→    94→Every entity includes `createdAt` (non-nullable, default `now()`) and `updatedAt` (non-nullable, default `now()`, auto-updated via `@updatedAt`). These serve audit, debugging, and analytics purposes.
    95→    95→
    96→    96→### 2.5 Financial Precision
    97→    97→
    98→    98→All monetary values use `Decimal` type (mapped to PostgreSQL `NUMERIC(19,4)`). This prevents floating-point rounding errors that would be unacceptable in financial calculations. Return rates use `Decimal` for precision (e.g., `0.50` for 0.5%).
