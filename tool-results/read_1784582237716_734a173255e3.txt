    28→## 1. Design Philosophy
    29→
    30→### 1.1 Dark Premium Aesthetic
    31→
    32→TeslaPrimeCapital's visual identity is rooted in a dark, premium aesthetic directly inspired by the Tesla automotive brand. Every surface, card, and interactive element is designed to communicate institutional-grade financial management. The dark palette is not a secondary or alternate theme — it is the singular, canonical visual expression of the platform. This choice eliminates theme-switching complexity and ensures pixel-perfect consistency across every viewport and device. The near-black backgrounds create a cinematic canvas that allows the red accent color to function as a powerful focal point, drawing the user's eye to calls-to-action, critical data points, and navigational indicators. The design language borrows from Tesla's own digital properties: expansive negative space, razor-sharp typography, and an unwavering commitment to minimal ornamentation. Every decorative element must earn its place; if it does not serve an informational or interactive purpose, it is removed.
    33→
    34→### 1.2 Trust-Signaling Through Professional Design
    35→
    36→As a managed investment plan platform, TeslaPrimeCapital must immediately establish credibility and trust with every visitor. The design system achieves this through several deliberate strategies. First, typographic hierarchy is strictly enforced — headings carry authoritative weight through the Poppins family at 600–700 weight, while body content remains highly legible via Rubik at 400–500 weight. Second, data presentation follows financial industry conventions: monospaced numerals for financial figures, consistent decimal alignment, and color-coded performance indicators (green for gains, red for losses, amber for warnings). Third, interactive elements provide immediate and unambiguous feedback — buttons respond to hover and press states, form fields validate inline with clear error messaging, and navigation elements maintain persistent active states. Fourth, generous whitespace and structured grid layouts convey a sense of order and meticulous organization, reinforcing the perception that the platform itself is as disciplined as the investment strategies it manages.
    37→
    38→### 1.3 Mobile-First Approach
    39→
    40→All layout and component decisions begin at the smallest viewport (320px) and progressively enhance upward. This methodology ensures that the core experience — viewing investment plans, checking portfolio performance, initiating transactions — is fully functional on mobile devices without reliance on desktop-specific patterns. Responsive behavior is not an afterthought applied via media query overrides; it is the foundational design methodology. Components are designed in their mobile configuration first, then adapted for tablet and desktop contexts. Navigation collapses into a hamburger menu on mobile, cards stack vertically, data tables transform into card-based layouts, and modals become full-screen overlays. The mobile-first approach also enforces strict touch target sizing (minimum 44×44px) and thumb-friendly placement of primary actions within the lower half of the viewport.
    41→
    42→### 1.4 Accessibility Standards (WCAG 2.1 AA)
    43→
    44→TeslaPrimeCapital targets WCAG 2.1 Level AA compliance as a minimum baseline. All text meets or exceeds the 4.5:1 contrast ratio requirement for normal text and 3:1 for large text (18px+ or 14px+ bold) against their respective backgrounds. Interactive elements provide visible focus indicators using a 2px solid ring with a 2px offset in the primary red color, ensuring keyboard navigability is not only functional but discoverable. Color is never used as the sole means of conveying information — status indicators always include accompanying icons or text labels. All images and icons carry appropriate ARIA attributes, and dynamic content updates (toasts, live portfolio values) leverage ARIA live regions. The `prefers-reduced-motion` media query is respected throughout, replacing animations with instant state changes for users who have enabled this setting at the OS level.

   141→## 3. Typography System
   142→
   143→### 3.1 Font Family Declarations
   144→
   145→The typography system employs three distinct font families, each serving a specific communicative role. All fonts are loaded from Google Fonts with `display=swap` to prevent layout shift.
   146→
   147→```css
   148→/* Headings — Poppins */
   149→--font-heading: 'Poppins', ui-sans-serif, system-ui, -apple-system, 'Segoe UI', sans-serif;
   150→
   151→/* Body Text — Rubik */
   152→--font-body: 'Rubik', ui-sans-serif, system-ui, -apple-system, 'Segoe UI', sans-serif;
   153→
   154→/* Editorial/Accent — Merriweather */
   155→--font-editorial: 'Merriweather', ui-serif, Georgia, 'Times New Roman', serif;
   156→
   157→/* Monospace/Code — JetBrains Mono (supplemental) */
   158→--font-mono: 'JetBrains Mono', ui-monospace, 'Cascadia Code', 'Fira Code', monospace;
   159→```

   206→
   207→### 3.5 Monospace / Code Font Specifications
   208→
   209→Monospace text uses JetBrains Mono (supplemented by system monospace fallbacks). It is reserved for financial figures, transaction IDs, code snippets, and any content requiring character-level alignment. All numerical displays use `font-variant-numeric: tabular-nums` to ensure consistent digit widths. The default size for inline code is `text-sm` with a background of `rgba(255,255,255,0.05)` and `rounded px-1.5 py-0.5` padding. Block code uses `text-sm` with `rounded-lg p-4 bg-gray-900` and `overflow-x-auto`.

   213→## 4. Spacing System
   214→
   215→### 4.1 Base Unit
   216→
   217→The spacing system is built on a **4px base unit**. All spacing values are multiples of this base unit, ensuring visual rhythm and consistency. Tailwind CSS's default spacing scale maps directly to this system, where `1` = 4px, `2` = 8px, `3` = 12px, and so on.
   218→
   219→### 4.2 Spacing Scale
   220→
   221→| Tailwind Token | Value (px/rem) | Common Usage |
   222→|---|---|---|
   223→| `space-0.5` / `p-0.5` / `m-0.5` | 2px | Tight icon-label gaps |
   224→| `space-1` / `p-1` / `m-1` | 4px | Inline spacing, icon padding |
   225→| `space-1.5` / `p-1.5` / `m-1.5` | 6px | Badge padding, small gaps |
   226→| `space-2` / `p-2` / `m-2` | 8px | Input inner padding, button padding (sm), list item gaps |
   227→| `space-2.5` | 10px | Tight form layouts |
   228→| `space-3` / `p-3` / `m-3` | 12px | Card padding (compact), group spacing |
   229→| `space-4` / `p-4` / `m-4` | 16px | Default card padding, form field gaps, section inner padding |
   230→| `space-5` | 20px | Comfortable form spacing |
   231→| `space-6` / `p-6` / `m-6` | 24px | Card padding (default), section gaps, modal padding |
   232→| `space-8` / `p-8` / `m-8` | 32px | Section vertical padding (mobile), large card padding |
   233→| `space-10` | 40px | Section gaps (desktop) |
   234→| `space-12` / `p-12` | 48px | Section vertical padding (desktop) |
   235→| `space-16` | 64px | Major section separators |
   236→| `space-20` | 80px | Page-level top/bottom padding |
   237→| `space-24` | 96px | Hero section padding |
   238→
   239→### 4.3 Component-Specific Spacing Rules
   240→
   241→**Cards:**
   242→- Default padding: `p-6` (24px) on all sides
   243→- Compact variant: `p-4` (16px)
   244→- Large/featured variant: `p-8` (32px)
   245→- Internal content gap: `space-y-4` (16px) between child elements
   246→- Card-to-card gap in grids: `gap-6` (24px)
   247→
   248→**Sections:**
   249→- Mobile vertical padding: `py-8` to `py-12` (32px–48px)
   250→- Desktop vertical padding: `py-12` to `py-20` (48px–80px)
   251→- Section title to content gap: `mt-6` (24px)
   252→- Between sections: `space-y-16` (64px) or `space-y-20` (80px) at page level
   253→
   254→**Form Fields:**
   255→- Label to input gap: `mt-1.5` (6px)
   256→- Input to input vertical gap: `space-y-4` (16px)
   257→- Input horizontal padding: `px-4` (16px)
   258→- Input vertical padding: `py-2.5` to `py-3` (10px–12px)
   259→- Error message to input gap: `mt-1.5` (6px)
   260→- Helper text to input gap: `mt-1` (4px)
   261→- Form group to form group: `space-y-6` (24px)
   262→- Submit button top margin: `mt-8` (32px)
   263→
   264→**Buttons:**
   265→- Small: `px-3 py-1.5` (12px horizontal, 6px vertical)
   266→- Default: `px-5 py-2.5` (20px horizontal, 10px vertical)
   267→- Large: `px-8 py-3.5` (32px horizontal, 14px vertical)
   268→- Icon-only (square): `p-2.5` (10px) for 24px icons, `p-2` (8px) for 20px icons
   269→- Button group gap: `gap-3` (12px)
   270→
   271→### 4.4 Margin Conventions
   272→
   273→- **Page-level margins:** Handled by container max-width; no horizontal margin on the body.
   274→- **Component margins:** Components should NOT export horizontal margins. Spacing between sibling components is the parent's responsibility via flex/grid gap utilities.
   275→- **Section margins:** Sections use vertical padding (`py-*`) rather than margins to maintain consistent background color coverage.
   276→- **Auto margins:** `mx-auto` is used exclusively for centering fixed-width containers and elements. It is not used for alignment within flex or grid contexts (prefer `justify-center`, `items-center`, or `text-center`).

   304→### 5.3 Border Color Rules
   305→
   306→All borders on the platform use white with reduced opacity as their base color. This approach ensures borders feel integrated with the dark background rather than appearing as stark, visible lines. The default border color is `border-white/10` (subtle). When an element requires more visual separation, `border-white/20` (medium) is applied. Active or focused elements use `border-white/30` (strong). Accent borders (selected tabs, highlighted cards) use `border-red-600`. Error states use `border-red-500`. Success states use `border-emerald-400`.

   507→## 8. Animation & Motion
   508→
   509→### 8.1 Design Principles for Motion
   510→
   511→Animations on TeslaPrimeCapital serve functional purposes: they provide feedback, indicate state changes, and guide attention. Decorative animation is kept to an absolute minimum. All motion must feel deliberate and fast — the platform should feel responsive and immediate. Animation duration never exceeds 300ms for interactive state changes. Page transitions and complex animations may extend to 500ms but should feel smooth and purposeful.
   512→
   513→### 8.2 Transition Durations
   514→
   515→| Token | Tailwind Class | Duration | Usage |
   516→|---|---|---|---|
   517→| `--duration-instant` | `duration-75` | 75ms | Micro-interactions, color changes |
   518→| `--duration-fast` | `duration-100` | 100ms | Ripple effects, small state changes |
   519→| `--duration-normal` | `duration-150` | 150ms | Hover transitions, color shifts |
   520→| `--duration-moderate` | `duration-200` | 200ms | Card hover, button press, dropdown open |
   521→| `--duration-slow` | `duration-300` | 300ms | Modal open/close, sidebar slide, page fade |
   522→| `--duration-slower` | Custom `duration-500` | 500ms | Complex page transitions, hero animations |
   523→
   524→### 8.3 Easing Functions
   525→
   526→| Token | CSS Value | Usage |
   527→|---|---|---|
   528→| `--ease-default` | `ease-in-out` | Standard transitions (color, opacity, background) |
   529→| `--ease-spring` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | Scale transforms (button press, card lift) |
   530→| `--ease-out` | `cubic-bezier(0, 0, 0.2, 1)` | Enter animations (fade in, slide in) |
   531→| `--ease-in` | `cubic-bezier(0.4, 0, 1, 1)` | Exit animations (fade out, slide out) |
   532→| `--ease-bounce` | `cubic-bezier(0.68, -0.55, 0.265, 1.55)` | Rare, only for playful micro-interactions |
   533→
   534→### 8.4 Hover and Focus Transitions
   535→
   536→All interactive elements include transition declarations for the properties they animate. Standard pattern:
   537→
   538→- **Buttons:** `transition-all duration-200 ease-in-out` — animates background-color, border-color, box-shadow, and transform
   539→- **Links and Nav Items:** `transition-colors duration-150` — animates color only
   540→- **Cards (interactive):** `transition-all duration-200 ease-out` — animates border-color, box-shadow, and transform
   541→- **Inputs:** `transition-colors duration-150, transition-shadow duration-150` — animates border-color and box-shadow independently
   542→- **Toggles and Switches:** `transition-colors duration-200` for track, `transition-transform duration-200` for thumb
   543→
   544→### 8.5 Loading Animations
   545→
   546→**Skeleton Shimmer:**
   547→Used for content loading placeholders. Applied to `div` elements shaped like the expected content.
   548→
   549→```css
   550→@keyframes shimmer {
   551→  0% { background-position: -200% 0; }
   552→  100% { background-position: 200% 0; }
   553→}
   554→
   555→.skeleton {
   556→  background: linear-gradient(
   557→    90deg,
   558→    #1F2937 25%,
   559→    #374151 37%,
   560→    #1F2937 63%
   561→  );
   562→  background-size: 200% 100%;
   563→  animation: shimmer 1.5s ease-in-out infinite;
   564→  border-radius: 8px;
   565→}
   566→```
   567→
   568→Tailwind utility: `animate-pulse bg-gray-700 rounded-lg` as a simpler alternative.
   569→
   570→**Spinner:**
   571→- **Small (16px):** Used inline within buttons or text. `border-2 border-white/20 border-t-white rounded-full w-4 h-4 animate-spin`
   572→- **Default (24px):** Used for page-level loading, card loading. `border-2 border-white/20 border-t-white rounded-full w-6 h-6 animate-spin`
   573→- **Large (40px):** Used for full-page loading states. `border-3 border-white/10 border-t-red-600 rounded-full w-10 h-10 animate-spin`
   574→
   575→**Progress Bar:**
   576→- Container: `h-1 bg-gray-700 rounded-full overflow-hidden`
   577→- Fill: `h-full bg-red-600 rounded-full transition-all duration-300 ease-out`
   578→
   579→### 8.6 Page Transition Approach
   580→
   581→Page transitions in Next.js use a subtle fade effect. On route change, the outgoing page fades to `opacity-0` over 150ms while the incoming page fades in from `opacity-0` to `opacity-1` over 200ms. No sliding or scaling is applied to page transitions — the content simply cross-fades, maintaining spatial stability and reducing motion sickness risk. The transition is achieved via Next.js middleware or a layout-level transition wrapper using CSS transitions on opacity with `pointer-events-none` during the transition period.

   585→## 9. Icon System
   586→
   587→### 9.1 Icon Library
   588→
   589→TeslaPrimeCapital uses **Lucide React** as its icon library. Lucide provides a comprehensive set of clean, consistent, and optically balanced icons that align well with the platform's minimal aesthetic. All icons are delivered as React components with full TypeScript support. Lucide icons use a 24×24 viewBox with a 2px default stroke, which is customizable per instance.
   590→
   591→Alternative considerations: Heroicons (very similar, also acceptable as a drop-in), Phosphor Icons (more stylistic variety). The decision for Lucide is based on its lightweight tree-shaking, consistent 2px stroke weight, and comprehensive financial/business icon coverage (trending-up, wallet, credit-card, shield, etc.).
   592→
   593→### 9.2 Icon Sizing Scale
   594→
   595→| Token | Tailwind Class | Size (px) | Usage |
   596→|---|---|---|---|
   597→| `--icon-xs` | `w-4 h-4` | 16px | Inline icons within text, badge icons, close buttons |
   598→| `--icon-sm` | `w-5 h-5` | 20px | Navigation icons (mobile), small action buttons, table action icons |
   599→| `--icon-md` | `w-6 h-6` | 24px | Default size — navigation (desktop), form field icons, card header icons |
   600→| `--icon-lg` | `w-8 h-8` | 32px | Feature icons, empty state illustrations, large action buttons |
   601→
   602→### 9.3 Stroke Width
   603→
   604→- **Default:** `strokeWidth={1.5}` — used for the majority of icons to maintain a refined, lightweight appearance
   605→- **Emphasis:** `strokeWidth={2}` — used for icons that need stronger visual weight, such as warning indicators, status icons in empty states, or icons displayed at small sizes (16px) where thinner strokes lose legibility
   606→- **Delicate:** `strokeWidth={1}` — rarely used, reserved for decorative icons in marketing contexts at large sizes (32px+)
   607→
   608→### 9.4 Color Inheritance Rules
   609→
   610→Icons inherit their color from the parent element's `color` CSS property via `currentColor`. This means:
   611→
   612→- Icons inside `text-white` containers render as white
   613→- Icons inside `text-gray-400` containers render as gray-400
   614→- Icons inside `text-red-500` containers render as red-500
   615→- Explicit color overrides: Apply `text-*` classes directly to the icon element when it needs a different color from its parent
   616→
   617→Icons should **never** use hardcoded `fill` or `stroke` color attributes. All coloring is handled through CSS `color` and the Lucide component's default behavior of applying `currentColor` to the stroke.
   618→
   619→### 9.5 Icon Usage Patterns
   620→
   621→- **Icon + Label:** `inline-flex items-center gap-2` with icon at `w-4 h-4` or `w-5 h-5` and text at `text-sm`
   622→- **Icon Button:** `p-2 rounded-lg hover:bg-white/5 transition-colors duration-150` with icon at `w-5 h-5`
   623→- **Feature Icon:** Centered in a circular or rounded-square container, `w-12 h-12 rounded-xl bg-red-600/10 flex items-center justify-center` with icon at `w-6 h-6 text-red-400`
   624→- **Empty State Icon:** Large centered icon (`w-12 h-12` or `w-16 h-16`) in `text-gray-600` with descriptive text below

   628→## 10. Responsive Breakpoints
   629→
   630→### 10.1 Breakpoint Definitions
   631→
   632→TeslaPrimeCapital uses Tailwind CSS's default breakpoint system. All responsive design begins at the base (mobile) and progressively enhances at each breakpoint.
   633→
   634→| Breakpoint | Tailwind Prefix | Min Width | Typical Device | Primary Layout Strategy |
   635→|---|---|---|---|---|
   636→| Base | (none) | 0px | Mobile phones (portrait) | Single column, stacked, full-width |
   637→| `sm` | `sm:` | 640px | Mobile phones (landscape), small tablets | Two-column grids begin, side-by-side pairs |
   638→| `md` | `md:` | 768px | Tablets (portrait), large phones | Standard tablet layout, sidebar consideration |
   639→| `lg` | `lg:` | 1024px | Tablets (landscape), small laptops | Desktop layouts activate, sidebar appears |
   640→| `xl` | `xl:` | 1280px | Laptops, desktops | Full desktop experience, multi-column |
   641→| `2xl` | `2xl:` | 1536px | Large desktops, external monitors | Max container width reached, wider spacing |
   642→
   643→### 10.2 Mobile-Specific Layout Rules
   644→
   645→- **Navigation:** Top bar with hamburger menu icon (`<` icon). Sidebar drawer slides in from the left with overlay.
   646→- **Cards:** Single column, full-width with `px-4` page padding. Internal card padding reduces to `p-4` (16px).
   647→- **Form Inputs:** Full-width (`w-full`), stacked vertically with `space-y-4`.
   648→- **Data Tables:** Hidden by default on mobile. Replaced by card-based list layout where each row becomes a card showing key fields. Alternatively, table is placed in horizontal scroll container with sticky first column.
   649→- **Buttons:** Full-width primary CTAs (`w-full`) for single-action forms. Inline buttons remain auto-width.
   650→- **Modals:** Full-screen overlay approach. Modal takes `min-h-screen w-full` with close button at top-right. Bottom sheet pattern is preferred for action sheets.
   651→- **Toasts:** Stacked at bottom of screen, full-width with `mx-4` margin.
   652→- **Typography:** h1 scales to `text-3xl`, h2 to `text-2xl`. No reduction below `text-sm` for body text to maintain readability.
   653→- **Hero Sections:** Full-width images/gradients, stacked content (heading → subtitle → CTA buttons stacked vertically), `py-12` vertical padding.
   654→
   655→### 10.3 Tablet Adjustments (md: 768px+)
   656→
   657→- **Navigation:** Horizontal top navigation bar may appear for simpler pages. Dashboard sidebar remains as drawer.
   658→- **Cards:** Two-column grid (`grid-cols-2`) for card collections. Card padding returns to `p-6` (24px).
   659→- **Form Layouts:** Two-column form grid for non-sequential fields (e.g., first name + last name side by side).
   660→- **Data Tables:** Horizontal scroll container with all columns visible. Sticky first column if table is wide.
   661→- **Modals:** Centered dialog with `max-w-lg` and rounded corners, no longer full-screen.
   662→- **Typography:** h1 scales to `text-4xl`, h2 to `text-3xl`.
   663→
   664→### 10.4 Desktop Multi-Column Layouts (lg: 1024px+)
   665→
   666→- **Navigation:** Horizontal top nav for marketing pages. Persistent sidebar for dashboard/app pages.
   667→- **Cards:** Three-column grid (`grid-cols-3`) for standard cards. Feature cards may use asymmetric layouts (8+4, 4+8).
   668→- **Form Layouts:** Sidebar form pattern — form fields on left (9 cols), summary/preview panel on right (3 cols).
   669→- **Data Tables:** Full table with all columns, hover rows, sortable headers. No horizontal scroll needed for standard tables.
   670→- **Modals:** Centered dialog, `max-w-lg` default, `max-w-2xl` for complex content.
   671→- **Spacing:** Section padding increases to `py-16` or `py-20`. Card-to-card gaps increase to `gap-6` or `gap-8`.

   675→## 11. Dark Theme Specification
   676→
   677→### 11.1 Dark-Only Architecture
   678→
   679→TeslaPrimeCapital uses a **dark-only** design approach. There is no light theme toggle, no `prefers-color-scheme` conditional, and no theme-switching mechanism. The dark aesthetic is the definitive visual identity of the platform. This architectural decision simplifies the codebase significantly — no CSS custom property theming layer, no flash-of-unstyled-content (FOUC) prevention, no theme context provider in React, and no theme persistence logic.
   680→
   681→All color values defined in this document are the final, production values applied globally. Tailwind's `darkMode` configuration is set to `'class'` as a safety mechanism, but the `dark` class is applied permanently at the root `<html>` element and never removed.
   682→
   683→### 11.2 Surface Elevation Hierarchy
   684→
   685→In a dark theme, elevation is communicated through **lightness** — higher-elevation surfaces are slightly lighter than their parent surfaces. This creates a visual stacking order without relying solely on shadows (which are less perceptible against dark backgrounds).
   686→
   687→| Elevation Level | Background Token | Tailwind Class | Lightness | Usage |
   688→|---|---|---|---|---|
   689→| Level 0 (Base) | `#000000` | `bg-black` | 0% | Page background, `<body>` |
   690→| Level 1 (Sunken) | `#0A0A0A` | `bg-[#0A0A0A]` | 4% | Subtle inset areas, footer background |
   691→| Level 2 (Default Surface) | `#111827` | `bg-gray-900` | 11% | Standard panels, sidebar, table header |
   692→| Level 3 (Raised) | `#1F2937` | `bg-gray-800` | 18% | Cards, inputs, dropdowns |
   693→| Level 4 (Overlay) | `#374151` | `bg-gray-700` | 27% | Hover states, pressed buttons, tooltips |
   694→| Level 5 (Modal) | `#1F2937` + `shadow-2xl` | `bg-gray-800 shadow-2xl` | 18% + shadow | Modals, popovers (shadow adds perceived elevation) |
   695→
   696→### 11.3 Text Contrast Ratios
   697→
   698→All text-surface color combinations have been verified against WCAG 2.1 AA contrast requirements.
   699→
   700→| Text Color | On Surface | Contrast Ratio | Passes AA Normal (4.5:1) | Passes AA Large (3:1) |
   701→|---|---|---|---|---|
   702→| `#FFFFFF` (white) | `#000000` (black) | 21:1 | Yes | Yes |
   703→| `#FFFFFF` (white) | `#0A0A0A` | 19.93:1 | Yes | Yes |
   704→| `#FFFFFF` (white) | `#111827` (gray-900) | 17.4:1 | Yes | Yes |
   705→| `#FFFFFF` (white) | `#1F2937` (gray-800) | 13.6:1 | Yes | Yes |
   706→| `#9CA3AF` (gray-400) | `#000000` (black) | 7.1:1 | Yes | Yes |
   707→| `#9CA3AF` (gray-400) | `#111827` (gray-900) | 5.9:1 | Yes | Yes |
   708→| `#9CA3AF` (gray-400) | `#1F2937` (gray-800) | 4.6:1 | Yes (barely) | Yes |
   709→| `#6B7280` (gray-500) | `#000000` (black) | 4.6:1 | Yes (barely) | Yes |
   710→| `#6B7280` (gray-500) | `#111827` (gray-900) | 3.8:1 | No | Yes |
   711→| `#6B7280` (gray-500) | `#1F2937` (gray-800) | 3.0:1 | No | Yes (barely) |
   712→| `#DC2626` (red-600) | `#000000` (black) | 5.5:1 | Yes | Yes |
   713→| `#DC2626` (red-600) | `#111827` (gray-900) | 4.6:1 | Yes (barely) | Yes |
   714→| `#10B981` (emerald-500) | `#000000` (black) | 8.8:1 | Yes | Yes |
   715→| `#F59E0B` (amber-500) | `#000000` (black) | 10.3:1 | Yes | Yes |
   716→
   717→**Critical Note:** `#6B7280` (gray-500) on `#1F2937` (gray-800) yields only a 3.0:1 contrast ratio. This combination is acceptable ONLY for large text (18px+ or 14px+ bold) or decorative/non-essential text. It must NOT be used for body text, labels, or any information-critical content on `gray-800` surfaces. For small text on `gray-800`, use `#9CA3AF` (gray-400) minimum.
   718→
   719→### 11.4 Visual Depth Techniques
   720→
   721→Beyond surface lightness, the dark theme employs additional techniques to communicate depth:
   722→
   723→- **Shadows:** All shadow definitions use black-based values (`rgba(0,0,0,*)`) with higher opacity than light themes (0.3–0.7 range). Shadows are more visible on dark surfaces when the shadow is darker than the surface.
   724→- **Borders:** Subtle white borders (`rgba(255,255,255,0.1)`) define edges where background lightness differences are insufficient.
   725→- **Glow Effects:** Red glow on accent elements creates a sense of energy and interactivity that pure elevation cannot convey.
   726→- **Backdrop Blur:** `backdrop-blur-sm` (4px) or `backdrop-blur` (8px) on overlay elements (modals, mobile nav) creates frosted-glass depth that reinforces z-axis layering.

   997→## 13. Accessibility Requirements
   998→
   999→### 13.1 Contrast Ratio Standards
  1000→
  1001→All text must meet WCAG 2.1 Level AA contrast requirements against its immediate background:
  1002→
  1003→- **Normal text (below 18px, or below 14px bold):** Minimum contrast ratio of **4.5:1**
  1004→- **Large text (18px+ or 14px+ bold):** Minimum contrast ratio of **3:1**
  1005→- **UI components and graphical objects:** Minimum contrast ratio of **3:1** against adjacent colors
  1006→- **Non-text contrast:** Borders, form field outlines, and icon separators must maintain at least 3:1 contrast against their background
  1007→
  1008→The contrast ratios for all text-surface combinations used in this design system are documented in Section 11.3 (Dark Theme Specification). Any new color combination introduced during implementation must be manually verified using a contrast checker tool before merging.
  1009→
  1010→### 13.2 Focus Indicators
  1011→
  1012→Focus indicators are critical for keyboard navigation. The design system enforces a consistent, highly visible focus ring:
  1013→
  1014→- **Style:** 2px solid ring in red-500 (`#EF4444`)
  1015→- **Offset:** 2px ring-offset in gray-900 (`#111827`) to ensure visibility against any background
  1016→- **Border-radius:** Inherits from the focused element's border-radius
  1017→- **CSS:** `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-red-500 focus-visible:ring-offset-2 focus-visible:ring-offset-gray-900`
  1018→- **Global Default:** Applied in the `@layer base` stylesheet so all focusable elements inherit this behavior unless explicitly overridden
  1019→- **Custom Components:** Custom components (buttons, links, inputs) may apply their own focus styles, but must maintain the same ring color and offset pattern for consistency
  1020→
  1021→The `focus-visible` pseudo-class is used instead of `:focus` to ensure focus indicators appear only during keyboard navigation, not on mouse clicks. This prevents visual noise for mouse users while preserving full accessibility for keyboard users.
  1022→
  1023→### 13.3 Touch Target Sizes
  1024→
  1025→All interactive elements must meet minimum touch target dimensions to ensure usability on touch devices:
  1026→
  1027→- **Minimum size:** 44×44 pixels (WCAG 2.5.5)
  1028→- **Measured size:** The touch target includes any visible padding around the clickable area. A 24px icon inside a 44px padded button meets the requirement.
  1029→- **Minimum spacing:** 8px between adjacent touch targets to prevent accidental activation
  1030→- **Inline links:** Text links within paragraphs are exempt from the 44px height requirement but should have generous line-height (`leading-relaxed` or `leading-7`) and surrounding whitespace
  1031→- **Form controls:** Checkboxes and radio buttons should have a minimum 44×44px tap area, which may be achieved via invisible padding on a wrapping `<label>` element
  1032→- **Implementation pattern:** Use `min-h-[44px] min-w-[44px]` Tailwind utilities on interactive elements, or ensure that `px-4 py-2.5` (which yields ~40px height for 16px text with 14px line-height, rounded up by the browser) is the minimum button padding
  1033→
  1034→### 13.4 Screen Reader Considerations
  1035→
  1036→- **Language:** The `<html>` element must declare `lang="en"` for proper screen reader pronunciation
  1037→- **Page titles:** Every page must have a unique, descriptive `<title>` tag following the pattern "Page Name | TeslaPrimeCapital"
  1038→- **Heading hierarchy:** Headings must follow a logical nesting order (h1 → h2 → h3, never skipping levels). Each page should have exactly one `<h1>`
  1039→- **ARIA landmarks:** The layout uses semantic HTML landmarks (`<header>`, `<nav>`, `<main>`, `<footer>`, `<aside>`, `<section>`) to provide navigation structure. Explicit `role` attributes are unnecessary when using semantic elements
  1040→- **ARIA labels:** Icon-only buttons must have `aria-label` describing their action (e.g., `aria-label="Close modal"`, `aria-label="Open navigation menu"`)
  1041→- **Live regions:** Dynamic content updates use `aria-live="polite"` for non-urgent updates (portfolio value changes) and `aria-live="assertive"` for critical notifications (error messages, transaction confirmations)
  1042→- **Form accessibility:** All form inputs must have associated `<label>` elements (using `htmlFor`/`id` pairing). Error messages must be associated via `aria-describedby`. Required fields must use `aria-required="true"`
  1043→- **Skip navigation:** A "Skip to main content" link must be present at the top of every page, visually hidden but accessible via keyboard (`Tab` key)
  1044→- **Modal focus trapping:** When a modal is open, focus must be trapped within the modal container. Tabbing must cycle through focusable elements inside the modal only. On close, focus must return to the element that triggered the modal
  1045→- **Image alt text:** Decorative images use `alt=""`. Informational images have descriptive alt text. Complex charts and graphs have extended descriptions via `aria-describedby` or adjacent visually-hidden text
  1046→
  1047→### 13.5 Reduced Motion Support
  1048→
  1049→The design system fully supports the `prefers-reduced-motion: reduce` media query. This setting is commonly enabled by users with vestibular disorders, motion sensitivity, or those who simply prefer less animation.
  1050→
  1051→- **Global rule:** When `prefers-reduced-motion: reduce` is active, all `animation-duration` and `transition-duration` values are set to `0.01ms`, and `animation-iteration-count` is set to `1`
  1052→- **Scroll behavior:** `scroll-behavior: auto` replaces `scroll-smooth` to prevent smooth scrolling animations
  1053→- **Skeleton loaders:** Replace shimmer animation with static `bg-gray-700` placeholder blocks
  1054→- **Spinners:** Replace `animate-spin` with a static progress indicator or text label ("Loading...")
  1055→- **Hover effects:** `transform` changes on hover (scale, translate) are disabled. Color and background transitions may remain at `duration-0` (instant)
  1056→- **Page transitions:** Disabled entirely; pages swap instantly
  1057→- **Implementation:** The global CSS in `app/globals.css` includes a `@media (prefers-reduced-motion: reduce)` block that handles the majority of cases. Component-level animations should also check this preference via `useReducedMotion()` hook (from `framer-motion` or custom implementation) when framework-level control is needed
  1058→
  1059→### 13.6 Keyboard Navigation
  1060→
  1061→- **Tab order:** Interactive elements must follow a logical left-to-right, top-to-bottom tab order that matches the visual layout
  1062→- **Focus management:** Modal dialogs must implement focus trapping. Dropdown menus must return focus to the trigger element on close. After page navigation, focus should move to the page's main heading or a skip-link target
  1063→- **Keyboard shortcuts:** If keyboard shortcuts are implemented, they must be documented in an accessible help panel. All keyboard shortcuts must use modifier keys (Ctrl/Cmd) to avoid conflicts with screen reader commands
  1064→- **Custom widgets:** Any custom interactive component (tabs, accordions, carousels) must follow WAI-ARIA Authoring Practices Guide (APG) patterns for the respective widget role
  1065→
