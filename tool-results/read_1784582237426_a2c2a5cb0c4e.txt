    42‚Üí## 1. Design Principles
    43‚Üí
    44‚Üí### 1.1 RESTful Conventions
    45‚Üí
    46‚Üí- Resources are modeled as **plural nouns** (`/investments`, not `/investment`).
    47‚Üí- HTTP methods convey intent: `GET` (read), `POST` (create), `PUT` (full update), `PATCH` (partial update), `DELETE` (remove).
    48‚Üí- All request and response bodies use `Content-Type: application/json`. File uploads use `multipart/form-data`.
    49‚Üí- Date/time values use **ISO 8601 UTC** (`2025-01-15T14:30:00.000Z`).
    50‚Üí- Monetary values are **decimal strings in USD** (`"amount": "1500.00"`) to avoid floating-point precision loss.
    51‚Üí- Resource identifiers are **UUIDs** (not auto-incrementing integers) for security and to prevent enumeration.
    52‚Üí
    53‚Üí### 1.2 URL-Based Versioning
    54‚Üí
    55‚ÜíAll endpoints are prefixed with `/api/v1/`. When breaking changes are required, a new version (`/api/v2/`) is deployed alongside the existing version with a minimum 6-month deprecation period. Deprecated endpoints include `Deprecation: true` and `Sunset: <date>` headers.
    56‚Üí
    57‚ÜíBackward-compatible changes (new response fields, new optional request fields, new endpoints) do not require a version increment.
    58‚Üí
    59‚Üí### 1.3 Pagination
    60‚Üí
    61‚ÜíTwo strategies are used based on data characteristics:
    62‚Üí
    63‚Üí| Strategy | Used For | Params | Navigation |
    64‚Üí|----------|----------|--------|------------|
    65‚Üí| **Offset-based** | Plans, admin user lists, tickets | `?page=1&limit=20` | Direct page access, `meta.total` included |
    66‚Üí| **Cursor-based** | Transactions, notifications, audit logs | `?cursor=<opaque>&limit=20` | Sequential only, `meta.hasMore` provided |
    67‚Üí
    68‚Üí- Default page size: **20** items.
    69‚Üí- Maximum page size: **100** items (requests exceeding this are silently capped, `X-Page-Size-Capped: true` header sent).
    70‚Üí- Cursor format: base64-encoded `{timestamp}_{uuid}` ‚Äî opaque to clients.
    71‚Üí
    72‚Üí### 1.4 Filtering & Sorting
    73‚Üí
    74‚Üí- **Filtering:** Query parameters prefixed with field names: `?status=active&planId=<uuid>`.
    75‚Üí- **Sorting:** `?sortBy=createdAt&sortOrder=desc`. Default sort is `createdAt` descending.
    76‚Üí- **Date range filtering:** `?startDate=2025-01-01&endDate=2025-06-30` (ISO date strings).
    77‚Üí- **Search:** `?search=john@example.com` ‚Äî performs partial match on supported fields (documented per endpoint).
    78‚Üí
    79‚Üí### 1.5 Stateless Design
    80‚Üí
    81‚ÜíEach request must contain all information needed to process it. The only stateful mechanism is the refresh token (HTTP-only cookie). No server-side session storage.
    82‚Üí
    83‚Üí### 1.6 HATEOAS-Lite Pagination Links
    84‚Üí
    85‚ÜíPaginated responses include `meta.prev` and `meta.next` URLs for navigation:
    86‚Üí
    87‚Üí```json
    88‚Üí{
    89‚Üí  "meta": {
    90‚Üí    "page": 1,
    91‚Üí    "limit": 20,
    92‚Üí    "total": 150,
    93‚Üí    "totalPages": 8,
    94‚Üí    "prev": "/api/v1/investments?page=0",
    95‚Üí    "next": "/api/v1/investments?page=2"
    96‚Üí  }
    97‚Üí}
    98‚Üí```

   102‚Üí## 2. Authentication & Authorization
   103‚Üí
   104‚Üí### 2.1 JWT Access Token
   105‚Üí
   106‚Üí- **Algorithm:** HS256
   107‚Üí- **Expiry:** 15 minutes
   108‚Üí- **Storage:** Frontend JavaScript memory only (module-level variable); never in localStorage, sessionStorage, or cookies.
   109‚Üí- **Header:** `Authorization: Bearer <access_token>`
   110‚Üí- **Payload (claims):**
   111‚Üí
   112‚Üí```typescript
   113‚Üíinterface JwtPayload {
   114‚Üí  sub: string;           // User UUID
   115‚Üí  role: Role;            // Current role
   116‚Üí  kycLevel: number;      // 0‚Äì3
   117‚Üí  activeMode: 'demo' | 'live';
   118‚Üí  tokenVersion: number;  // Incremented on password reset / forced logout
   119‚Üí  iat: number;           // Issued at (Unix timestamp)
   120‚Üí  exp: number;           // Expiration (Unix timestamp)
   121‚Üí}
   122‚Üí
   123‚Üítype Role = 'SUPER_ADMIN' | 'ADMIN' | 'COMPLIANCE' | 'SUPPORT' | 'USER';
   124‚Üí```
   125‚Üí
   126‚Üí### 2.2 Refresh Token
   127‚Üí
   128‚Üí- **Type:** Opaque random string (not a JWT)
   129‚Üí- **Expiry:** 7 days
   130‚Üí- **Storage:** HTTP-only, `Secure`, `SameSite=Strict` cookie named `__tpc_rt`
   131‚Üí- **Rotation:** Each refresh invalidates the old token and issues a new one (prevents replay attacks).
   132‚Üí- **Family tracking:** All tokens in a rotation chain are stored; if any token in the chain is reused after rotation, the entire family is revoked (theft detection).
   133‚Üí
   134‚Üí### 2.3 Two-Factor Authentication (2FA)
   135‚Üí
   136‚ÜíWhen 2FA is enabled for a user, the login flow is two-step:
   137‚Üí
   138‚Üí1. `POST /api/v1/auth/login` returns `HTTP 200` with `requires2FA: true` and a temporary `twoFactorToken` (short-lived, 5 minutes).
   139‚Üí2. Client submits `POST /api/v1/auth/verify-2fa` with the temporary token and TOTP code.
   140‚Üí3. On success, access + refresh tokens are issued.
   141‚Üí
   142‚Üí**2FA Verification Header:** For endpoints requiring confirmed 2FA (e.g., withdrawals), the client must include `X-2FA-Code: <6-digit TOTP>` to prove possession. This is validated server-side against the user's TOTP secret.
   143‚Üí
   144‚Üí### 2.4 Role Hierarchy & Middleware
   145‚Üí
   146‚Üí```
   147‚ÜíSUPER_ADMIN > ADMIN > COMPLIANCE > SUPPORT > USER
   148‚Üí```
   149‚Üí
   150‚ÜíHigher roles inherit all permissions from roles below them. Middleware checks are enforced at both the route level and endpoint level (defense in depth). Failed permission checks return `403 Forbidden`.
   151‚Üí
   152‚Üí### 2.5 Public vs. Protected Endpoints
   153‚Üí
   154‚Üí| Category | Endpoints |
   155‚Üí|----------|-----------|
   156‚Üí| **Public** (no auth) | `register`, `login`, `forgot-password`, `reset-password`, `verify-email`, `resend-verification`, plan list, currencies, maintenance status |
   157‚Üí| **Protected** (auth required) | All other endpoints |
   158‚Üí
   159‚Üí### 2.6 Token Versioning
   160‚Üí
   161‚ÜíEach user has a `tokenVersion` in the database. Password resets and admin-forced logouts increment this value. Tokens containing a stale `tokenVersion` are rejected ‚Äî no blacklist needed.
   162‚Üí

   285‚Üí## 5. Rate Limiting Strategy
   286‚Üí
   287‚ÜíRate limiting is implemented via **Redis sliding-window counters**.
   288‚Üí
   289‚Üí### 5.1 Rate Limit Tiers
   290‚Üí
   291‚Üí| Endpoint Category | Limit | Window | Scope |
   292‚Üí|-------------------|-------|--------|-------|
   293‚Üí| **Auth (login, forgot-password, reset-password, verify-2fa)** | 5 requests | 1 minute | Per IP |
   294‚Üí| **Auth (register, verify-email, resend-verification)** | 3 requests | 1 minute | Per IP |
   295‚Üí| **Auth (refresh-token)** | 30 requests | 1 minute | Per IP |
   296‚Üí| **Write (create deposit, withdrawal, investment)** | 10 requests | 1 minute | Per User |
   297‚Üí| **KYC submit** | 3 requests | 1 hour | Per User |
   298‚Üí| **Support ticket create** | 5 requests | 1 hour | Per User |
   299‚Üí| **Read (list, detail, dashboard)** | 60 requests | 1 minute | Per User |
   300‚Üí| **Admin (all admin endpoints)** | 120 requests | 1 minute | Per User |
   301‚Üí| **File upload (signed URL generation)** | 10 requests | 1 minute | Per User |
   302‚Üí
   303‚Üí### 5.2 Response Headers (Every Request)
   304‚Üí
   305‚Üí| Header | Description |
   306‚Üí|--------|-------------|
   307‚Üí| `X-RateLimit-Limit` | Max requests allowed in the current window |
   308‚Üí| `X-RateLimit-Remaining` | Requests remaining in the current window |
   309‚Üí| `X-RateLimit-Reset` | Unix timestamp when the window resets |
   310‚Üí
   311‚Üí### 5.3 Rate Limit Exceeded Response
   312‚Üí
   313‚Üí```
   314‚ÜíHTTP 429 Too Many Requests
   315‚ÜíRetry-After: 45
   316‚Üí```
   317‚Üí
   318‚Üí```json
   319‚Üí{
   320‚Üí  "success": false,
   321‚Üí  "message": "Rate limit exceeded. Please retry after 45 seconds.",
   322‚Üí  "code": "RATE_LIMIT_EXCEEDED"
   323‚Üí}
   324‚Üí```

   328‚Üí## 6. Domain: Authentication
   329‚Üí

   331‚Üí
   332‚Üí### 6.1 Register
   333‚Üí
   334‚Üí```
   335‚ÜíPOST /api/v1/auth/register
   336‚Üí```
   337‚Üí
   338‚Üí**Auth:** None (public)
   339‚Üí
   340‚Üí**Description:** Register a new user account. Creates the user in `pending_verification` state, generates an email verification token (24h expiry), sends a verification email, and credits the Demo wallet with the configured starting balance upon email verification.
   341‚Üí
   342‚Üí**Request Body:**
   343‚Üí
   344‚Üí```typescript
   345‚Üíinterface RegisterRequest {
   346‚Üí  email: string;           // Valid email, max 254 chars
   347‚Üí  password: string;        // Min 8 chars, 1 uppercase, 1 lowercase, 1 number
   348‚Üí  confirmPassword: string; // Must match password
   349‚Üí  referralCode?: string;   // Optional referral code from existing user
   350‚Üí}
   351‚Üí```
   352‚Üí
   353‚Üí**Response (200):**
   354‚Üí
   355‚Üí```typescript
   356‚Üíinterface RegisterResponse {
   357‚Üí  user: {
   358‚Üí    id: string;            // UUID
   359‚Üí    email: string;
   360‚Üí    status: 'pending_verification';
   361‚Üí    createdAt: string;     // ISO 8601
   362‚Üí  };
   363‚Üí  message: string;         // "Verification email sent. Please check your inbox."
   364‚Üí}
   365‚Üí```
   366‚Üí
   367‚Üí**Status Codes:** `200` (success), `400` (validation), `409` (email exists, invalid referral code), `429` (rate limit)
   368‚Üí

   370‚Üí
   371‚Üí### 6.2 Login
   372‚Üí
   373‚Üí```
   374‚ÜíPOST /api/v1/auth/login
   375‚Üí```
   376‚Üí
   377‚Üí**Auth:** None (public)
   378‚Üí
   379‚Üí**Description:** Authenticate with email and password. If 2FA is enabled, returns `requires2FA: true` and a temporary token instead of access/refresh tokens. After 5 consecutive failed attempts, the account is locked for 15 minutes.
   380‚Üí
   381‚Üí**Request Body:**
   382‚Üí
   383‚Üí```typescript
   384‚Üíinterface LoginRequest {
   385‚Üí  email: string;
   386‚Üí  password: string;
   387‚Üí}
   388‚Üí```
   389‚Üí
   390‚Üí**Response (200) ‚Äî No 2FA:**
   391‚Üí
   392‚Üí```typescript
   393‚Üíinterface LoginResponse {
   394‚Üí  user: {
   395‚Üí    id: string;
   396‚Üí    email: string;
   397‚Üí    role: Role;
   398‚Üí    kycLevel: number;
   399‚Üí    activeMode: 'demo' | 'live';
   400‚Üí    twoFactorEnabled: boolean;
   401‚Üí  };
   402‚Üí  accessToken: string;     // JWT, 15min
   403‚Üí  message: string;
   404‚Üí}
   405‚Üí```
   406‚Üí
   407‚Üí**Response (200) ‚Äî 2FA Required:**
   408‚Üí
   409‚Üí```typescript
   410‚Üíinterface Login2FAResponse {
   411‚Üí  requires2FA: true;
   412‚Üí  twoFactorToken: string;  // Temporary token, 5min expiry
   413‚Üí  message: string;         // "Two-factor authentication required."
   414‚Üí}
