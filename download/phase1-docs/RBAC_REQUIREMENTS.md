# Role-Based Access Control (RBAC) Requirements

## 1. Role-Based Access Control Overview

The platform implements a hierarchical Role-Based Access Control (RBAC) system that governs all access to platform resources, APIs, and administrative functions. Every user account is assigned exactly one role at any given time, and that role determines the complete set of permissions available to that user. The hierarchy is strictly enforced: higher roles inherit all permissions from the roles below them in the chain, while also gaining exclusive capabilities that are reserved for their tier. This model simplifies permission management, ensures consistent security enforcement, and provides clear audit trails for all privileged operations. Role assignments and changes are logged immutably in the audit system, and notifications are sent to affected users whenever their role is modified. Admins with sufficient privileges can upgrade or downgrade user roles through a dedicated management interface, with restrictions on who can modify roles at each tier level.

## 2. Role Definitions

### Super Admin

The Super Admin role represents the highest privilege level in the system, granted only to platform owners and core technical staff. Super Admins have unrestricted access to every module, dataset, and configuration option within the platform. They can create, modify, and deactivate other admin accounts, assign and revoke roles at every level, and access the full system configuration panel including database settings, fee structures, and deployment controls. Super Admins can view aggregated financial reports, audit logs, and all user data across both demo and live modes. This role also has the authority to override automated security controls in exceptional circumstances, with all such overrides recorded in the audit trail. The number of Super Admin accounts should be kept to an absolute minimum, typically two to three individuals, to reduce the attack surface.

### Admin

The Admin role provides broad operational access to manage day-to-day platform activities without exposing critical system configuration. Admins can perform full user lifecycle management including creating accounts, updating profiles, banning users, and adjusting account statuses. They are responsible for processing KYC submissions, approving or rejecting deposit and withdrawal requests across both cryptocurrency and gift card channels, and managing the investment plan configurations. Admins have access to the reports and analytics dashboard to monitor platform performance, user growth, and transaction volumes. They can view the referral tree structure, adjust commission rates within approved ranges, and resolve referral disputes. Admins cannot modify system-level settings, manage other admins, or access the infrastructure configuration panel. All financial operations performed by Admins require dual-approval for amounts exceeding $10,000 in a single transaction.

### KYC Officer

The KYC Officer role is a specialized position focused exclusively on identity verification and compliance operations. KYC Officers can access the KYC review queue, view uploaded identity documents in full resolution, and approve or reject verification submissions at all levels (Basic, Standard, Premium). They can add review notes and rejection reasons that are communicated to users, and they can flag accounts for enhanced due diligence when documents appear suspicious or inconsistent. KYC Officers have read-only access to user profiles necessary for verification purposes but cannot modify user data, process financial transactions, or access support ticket systems. This role is critical for maintaining regulatory compliance with AML/CFT standards and ensuring that only verified users can access deposit and withdrawal capabilities. KYC Officers are bound by stricter audit logging requirements, and every document review action is recorded with timestamps and reviewer identity.

### Support Agent

The Support Agent role provides front-line customer assistance capabilities with deliberately limited data access to protect user privacy and financial information. Support Agents can view user profiles including name, email, verification level, and account status, but they cannot see financial balances, transaction details beyond the last five transactions, or KYC documents. They have full access to the support ticket system including the ability to view, respond to, escalate, and close tickets. Support Agents can view limited transaction history for troubleshooting purposes but cannot initiate or approve any financial operations. This role is designed to handle the majority of user inquiries while preventing unauthorized access to sensitive financial data. Support Agents cannot modify user roles, access administrative reports, or perform any system configuration changes.

### User (Investor)

The User role is the standard account type for all registered investors on the platform. Users can manage their own investments by browsing available plans, depositing funds through supported payment methods, selecting and activating investment plans, and monitoring their active and completed investments. They can request withdrawals of available wallet balances, manage their referral network by sharing referral links and tracking commission earnings, and update their own profile information including security settings like password changes and two-factor authentication. Users have full visibility into their own transaction history, investment performance, referral tree, and earnings breakdown. All financial operations are constrained to the user's own account, and the system enforces plan-tier eligibility based on the user's deposit amount and KYC verification level. Users operate in both demo and live modes, with a clear visual distinction between the two environments.

### Unverified User

The Unverified User role is the initial state for all newly registered accounts before email verification is completed. Users in this role can access their profile page and complete the email verification process, but all other platform functionality is restricted. They can browse the public-facing plan information and platform features in read-only mode, and they can interact with the demo mode to explore the platform's investment mechanics without risk. Unverified Users cannot deposit real funds, activate investment plans, initiate withdrawals, or access the referral system. This role exists as a security measure to prevent abuse from unverified registrations and to ensure that all accounts have a confirmed, reachable email address before any financial activity is permitted.

## 3. Permission Categories

### User Management

User Management permissions cover the full lifecycle of user accounts from creation through deletion. The **create** permission allows administrators to manually create user accounts, typically used for onboarding institutional investors or resolving registration issues. The **read** permission grants access to user profiles, account details, and activity history, with scope varying by role. The **update** permission enables modification of user profile data, account status, and role assignments. The **delete** permission allows permanent removal of user accounts and associated data, subject to regulatory retention requirements. The **ban/unban** permission enables administrators to restrict or restore user access to the platform, typically used for policy violations or suspicious activity. Each of these actions triggers an audit log entry and, where applicable, a notification to the affected user.

### KYC Management

KYC Management permissions govern access to the identity verification pipeline. The **review** permission allows access to the KYC submission queue and the ability to open individual verification cases for detailed examination. The **approve** permission authorizes the officer to confirm that submitted documents meet verification requirements and upgrade the user's verification level accordingly. The **reject** permission enables the officer to deny a verification submission with a mandatory reason that is communicated to the user. The **view documents** permission grants access to the uploaded identity documents stored in Cloudinary's private folder, including high-resolution originals. These permissions are exclusively assigned to KYC Officers and above, and every document access event is logged with the viewer's identity, timestamp, and the specific document accessed.

### Financial Operations

Financial Operations permissions are the most sensitive category in the system, controlling access to money movement and investment management. The **approve deposits** permission allows an administrator to verify incoming cryptocurrency transactions or gift card submissions and credit the user's wallet. The **approve withdrawals** permission enables verification and processing of withdrawal requests, including confirming sufficient balance and compliance with withdrawal limits. The **adjust balances** permission is a high-privilege operation that allows manual correction of user wallet balances, typically reserved for dispute resolution or error correction, and requires Super Admin authorization. The **manage plans** permission covers creating, modifying, and deactivating investment plan configurations including tier parameters, return rates, and duration settings. All financial operations are protected by additional controls including amount thresholds requiring dual approval and mandatory audit logging.

### Referral Management

Referral Management permissions control access to the platform's multi-tier referral and commission system. The **view referral tree** permission allows administrators to inspect the hierarchical referral structure, trace commission flows, and identify potential referral fraud patterns such as self-referral or circular referral chains. The **adjust commissions** permission enables correction of commission calculations in cases of errors or disputes, with all adjustments requiring detailed justification recorded in the audit log. The **resolve disputes** permission authorizes administrators to investigate and resolve referral-related conflicts between users, including cases where commission attribution is contested. These permissions help maintain the integrity of the referral program and ensure that commission payouts are accurate and fair.

### Support

Support permissions govern access to the customer support ticketing system. The **view tickets** permission grants access to the ticket queue and individual ticket details, with scope determined by the agent's role and any assigned territories or user segments. The **respond to tickets** permission enables agents to compose and send replies to user-submitted tickets, including attaching internal notes visible only to staff. The **close tickets** permission allows agents to mark tickets as resolved when the user's issue has been addressed. The **escalate** permission enables agents to route tickets to higher-tier support staff or administrative personnel when the issue requires elevated access or authority. All ticket interactions are tracked with response time metrics for service level monitoring.

### Reports

Reports permissions control access to the platform's analytics and reporting capabilities. The **view reports** permission grants access to pre-built report dashboards covering user growth, transaction volumes, revenue metrics, and investment performance. The **export reports** permission allows administrators to download report data in CSV, PDF, or Excel formats for external analysis or compliance reporting. The **analytics dashboard** permission provides access to real-time operational metrics including active sessions, pending transactions, system health indicators, and financial summaries. Report access is scoped by role, with Super Admins seeing platform-wide data and other roles seeing data relevant to their operational domain. Sensitive financial reports require Admin-level access or above.

### System Configuration

System Configuration permissions are the most privileged category, controlling the platform's operational parameters and structural settings. The **manage plans** permission covers creating, editing, and deactivating investment plan tiers, adjusting return rates, duration periods, and deposit ranges. The **manage fees** permission allows modification of the platform's fee structure including the 21% withdrawal fee and any other applicable charges. The **manage settings** permission provides access to general platform settings including maintenance modes, feature flags, and operational parameters. The **manage roles** permission is the most sensitive configuration permission, allowing the creation and modification of RBAC roles, permission assignments, and role hierarchies. These permissions are restricted to Super Admin level to prevent unauthorized configuration changes that could affect platform economics or security.

### Audit

Audit permissions govern access to the platform's immutable audit logging system. The **view audit logs** permission grants read access to the comprehensive log of all administrative and financial operations, filterable by user, action type, date range, and entity. The **export logs** permission allows downloading audit data for compliance reporting, internal reviews, or regulatory submissions. The **manage retention** permission controls the retention policy for audit logs, including setting retention periods for different log categories and managing the archival of older records. Audit logs are append-only and cannot be modified or deleted by any role, ensuring a complete and trustworthy record of all platform activity. Access to audit logs is restricted to Super Admin and Admin roles, with KYC Officers having limited read access to logs related to their own verification actions.

## 4. Permission Inheritance

The RBAC hierarchy follows a strict linear chain where each role inherits all permissions from the roles positioned below it: **Super Admin > Admin > KYC Officer > Support Agent > User**. This means a Super Admin automatically possesses every permission granted to Admins, KYC Officers, Support Agents, and Users, in addition to their exclusive Super Admin permissions. An Admin inherits all KYC Officer and Support Agent permissions, giving them the ability to review KYC submissions and handle support tickets in addition to their broader operational capabilities. A KYC Officer inherits Support Agent permissions, allowing them to assist with user inquiries that arise during the verification process. A Support Agent inherits User permissions, ensuring they can understand the platform from the user's perspective. This inheritance model eliminates permission duplication, simplifies role management, and ensures that promoting a user to a higher role never results in a loss of capability. The inheritance chain is enforced at the API middleware level, where a single permission check confirms access without needing to evaluate the full role hierarchy on each request.

## 5. Role Assignment Rules

Role assignments follow strict governance rules to prevent privilege escalation and maintain accountability. Only Super Admins can assign or revoke the Super Admin and Admin roles, ensuring that high-privilege accounts are controlled by the most trusted individuals. Admins can assign the KYC Officer and Support Agent roles to operational staff members. Admins can also upgrade verified Users to higher verification levels through the KYC process, but role changes between User, KYC Officer, Support Agent, and Admin require explicit administrative action. The Unverified User role is automatically assigned upon registration and is upgraded to User upon email verification; this transition is system-driven and does not require administrative intervention. Every role assignment or revocation is recorded in the audit log with the acting administrator's identity, the target user, the previous role, the new role, the timestamp, and an optional justification field. Affected users receive an in-app notification and email confirmation whenever their role changes, unless the change is a system-driven verification upgrade. Role changes take effect immediately upon save, but active sessions may require re-authentication to pick up the new permission set.

## 6. Data Access Scope

Each role operates within a defined data access scope that determines which records and entities they can view and modify. **Users** have access exclusively to their own data, including their profile, transactions, investments, referrals, and notifications. They cannot view any information belonging to other users. **Support Agents** can view profile information and limited transaction history for users who have open support tickets assigned to them, but cannot access financial balances, KYC documents, or data for users without active tickets. **KYC Officers** can view user profiles and KYC documents for all users with pending or in-review verification submissions, plus historical verification records for previously verified users. **Admins** have broad access to all user data across the platform, including full financial histories, KYC documents, referral structures, and support interactions. Admins can view data in aggregate through reports and individually through user profile pages. **Super Admins** have unrestricted access to all data in the system, including audit logs, system configuration history, and all user data regardless of status. Additionally, certain data dimensions have independent scope controls: demo mode data and live mode data are logically separated, and the system enforces that financial operations in demo mode cannot affect live mode balances or records. Admin-level and above roles can toggle between demo and live data views, while Users naturally see only their own data across both modes.
